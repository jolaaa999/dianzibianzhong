电子编钟交互演示系统

产品需求文档（PRD）

方案一：低成本视觉追踪

版本：v1.0 | 日期：2026-07-08 | 状态：草案

1. 需求背景

1.1 项目概述

本项目旨在开发一套电子编钟交互演示系统，用于室内展厅场景。体验者手持两根带有反光标记的敲击棒，在摄像头前方以真实编钟演奏的方式击打虚拟编钟，系统实时识别敲击位置并播放对应的编钟音色。系统需支持单音敲击与多音同时敲击两种演奏模式，还原真实编钟"金声玉振"的听觉体验。

1.2 方案选择

经过技术评估，本项目采用方案一：低成本视觉追踪方案，核心硬件为一台 USB
高清摄像头与贴于敲击棒顶端的反光标记小球。该方案利用 OpenCV
视觉算法实时识别反光标记的空间位置，输出 XY 坐标至前端展示界面。相比 IMU
惯性测量方案，视觉追踪无累积漂移问题，定位精度更高，且硬件成本极低。

1.4 用户与场景

目标用户为展厅参观者，涵盖不同年龄层，无需任何音乐基础即可上手体验。核心场景如下：

1.  参观者进入交互区域，拿起敲击棒；

2.  屏幕显示虚拟编钟阵列，用户根据提示或自由发挥进行敲击；

3.  单棒敲击单个编钟，发出独立音色；

4.  双棒同时敲击两个或多个编钟，发出和声/复合音色；

5.  系统给予即时视觉与听觉反馈，营造沉浸式文化体验。

1.5 痛点与需求

现有展厅互动装置多采用触摸屏或按钮交互，缺乏真实乐器的物理临场感。本项目的核心诉求是将编钟这一中国传统乐器以数字化方式还原，让体验者通过真实的肢体动作（挥棒敲击）触发声音，而非简单的点击屏幕。此外，编钟作为多音乐器，必须准确处理多音同时触发的复杂场景——多个编钟被同时敲击时，系统应发出融合后的复合音响，而非简单的单音叠加。

3.2 模块详细设计

3.2.1 视觉追踪系统

业务逻辑：摄像头以固定帧率（建议 ≥
30fps）采集画面。每帧图像经过灰度转换、阈值分割、轮廓检测等处理，筛选出符合反光标记球特征的连通区域。对每个有效标记球分配唯一追踪
ID，持续输出其中心点像素坐标。

交互逻辑：系统启动后自动枚举可用摄像头 → 进入实时预览画面 →
操作者挥动敲击棒 → 画面中实时叠加标记球位置指示器。

规则约束：

- 系统需同时追踪 2 个反光标记球（对应左右两根敲击棒）；

- 当画面中标记球数量 ≠ 2
  时，系统进入降级模式：仅追踪可见的球，缺失的球在界面上显示"离屏"状态；

- 标记球最小可识别直径：约 8 像素（取决于摄像头分辨率与拍摄距离）。

3.2.2 敲击检测引擎

业务逻辑：敲击检测是连接视觉坐标与声音触发的关键模块。系统维护每个标记球最近
N 帧（建议
N=5~8）的坐标历史，计算瞬时速度向量。当速度从正向（朝向屏幕/编钟方向）骤降至接近零，且此时坐标落在某个编钟的虚拟触发区域内，则判定为一次有效敲击。

交互逻辑：标记球进入编钟触发区域 → 若速度曲线符合"撞击"特征 →
触发该编钟发声 + 视觉高亮反馈。

规则约束：

- 每个编钟的触发区域为矩形/椭圆形碰撞盒，尺寸比编钟视觉区域略大（提供容错）；

- 连续两次敲击同一编钟的最小间隔（防重敲冷却时间）：80ms；

- 判定敲击的最低速度阈值：[Assumption — to be confirmed] 建议 50
  像素/帧；

- 双棒同时敲击的判定窗口：[Assumption — to be confirmed] 建议 ±50ms
  内触发的多个敲击视为"同时"。

3.2.3 编钟展示界面

业务逻辑：前端界面以半弧形阵列呈现虚拟编钟。编钟分为主编钟（音域核心，体积较大）与辅编钟（扩展音域，体积较小）两类。每个编钟对应一个独立音高。界面需实时响应视觉追踪数据，在棒子靠近时显示辅助瞄准提示，在敲击命中时播放动画。

规则约束：

- 编钟数量：[Assumption — to be confirmed] 建议 8~12 个（含 4~5
  个主编钟 + 4~7 个辅编钟）；

- 编钟音高排列遵循真实编钟规律：低音大钟在左（或前排），高音小钟在右（或后排）；

- 每个编钟的触发区域在界面上不可见（隐性碰撞盒），但在调试模式下可叠加显示；

- 敲击反馈动画持续时间：200ms，包含高亮（金色光晕）+
  轻微位移（模拟振动）。

3.2.4 声音合成引擎

业务逻辑：声音引擎负责编钟音色的存储、调度和混音。每个编钟对应一段预录制的高质量编钟采样音频（建议
WAV
或无损压缩格式）。系统维护一个活跃的"发声实例"列表，每个实例对应一次敲击触发。当多个实例同时活跃时，引擎在音频输出层级进行实时混音。

单音 vs 多音差异处理

  ---------------- -------------------------------------------------------------------- ----------------------------------
  场景             处理方式                                                             预期听觉效果

  单音敲击         直接播放对应编钟的采样音频，完整衰减周期（约 3~5 秒）                清澈、悠长的单音，带自然泛音

  双音同时         两个采样在同一音频帧启动，各自独立衰减，总音量进行压缩限幅防止削波   两音和谐共鸣，音量自然平衡

  三音及以上       多采样同时启动，启用轻度压缩 + 微量混响，模拟厅堂声学环境            丰满的和声效果，类似真实编钟合奏

  快速连续敲击     新实例独立启动，旧实例继续衰减（支持多实例叠加）                     颗粒感清晰的快速乐句
  ---------------- -------------------------------------------------------------------- ----------------------------------

规则约束：

- 同时活跃的最大发声实例数：≥ 8 个；

- 音频输出采样率：44.1kHz 或 48kHz；

- 单音到多音的音量补偿：每增加一个同时发声的音，整体增益衰减约
  3dB，防止总音量过载；

- 支持编钟音色的动态替换（便于后续更换音源素材）。

3.2.5 通信中间件

业务逻辑：视觉追踪模块（通常在 Python
进程中运行）与前端展示界面之间需要低延迟双向通信。追踪模块以固定频率（建议
30~60Hz）广播标记球坐标数据；前端在检测到有效敲击时，可选择性地回传确认信息用于调试。

规则约束：

- 通信方式：[Assumption — to be confirmed]
  WebSocket（推荐，便于网页端接入）或串口通信（若前端为桌面应用且要求物理隔离）；

- 单条消息结构：JSON 格式，包含 timestamp、stick_id、x、y、confidence；

- 网络中断恢复：自动重连，重连期间前端显示"追踪信号丢失"提示；

- 消息投递延迟：局域网/WebSocket 模式下 ≤ 20ms。

3.2.6 系统校准向导

业务逻辑：校准向导在系统首次启动或检测到硬件变更时自动弹出。流程分为三步：（1）摄像头画面确认——确保编钟阵列完整出现在画面内；（2）标记球识别测试——要求用户依次挥动左右敲击棒，系统确认两个标记球均被稳定识别；（3）触发映射验证——依次高亮每个编钟，用户用棒子敲击对应位置，系统记录并微调触发区域。

交互逻辑：向导采用步骤条形式，每完成一步解锁下一步。全程提供图文引导，预计
1~2 分钟完成。

3.2.7 演示模式控制

业务逻辑：系统具备两种全局状态：待机模式与演奏模式。待机模式下循环播放编钟文化介绍动画与吸引性视觉效果；当检测到标记球进入交互区域（或感应器检测到人），自动切换至演奏模式。连续
60 秒无敲击动作，自动返回待机模式。

5. 技术架构

系统采用分层设计，从下到上依次为硬件层（摄像头、敲击棒、显示屏）、追踪层（OpenCV
视觉识别）、通信层（实时数据传输）与应用层（展示界面、敲击检测、声音合成、校准向导）。各层之间通过明确定义的数据接口解耦，便于后续替换或升级单个模块。

硬件层：USB
高清摄像头采集画面，敲击棒顶端贴反光标记球，展厅显示屏输出前端界面。

追踪层：OpenCV 视觉追踪模块负责图像预处理、反光球检测、轮廓筛选、追踪 ID
分配与坐标输出。

通信层：WebSocket 或串口负责追踪模块与前端之间的实时坐标数据推送。

应用层：编钟展示界面渲染虚拟编钟阵列；敲击检测引擎根据坐标历史判定敲击动作；声音合成引擎管理音色播放与混音；校准向导引导用户完成系统初始化。
